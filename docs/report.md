# Mac开源语音输入法深度调研报告：基于Rime + 本地大模型的智能语音输入方案

## TL;DR（核心结论）

通过深度调研市面上**6个主流开源语音输入项目**和**3个核心ASR引擎**，确认 **Rime（鼠须管）+ Qwen3-ASR + MLX框架** 的组合是满足你所有需求的最佳技术路径。Rime作为macOS上最成熟的输入法引擎，支持全拼/双拼/辅助码等所有输入方案，通过`librime-lua`插件系统可实现深度扩展；Qwen3-ASR 0.6B（4-bit量化后仅**~500MB**）可在32GB Mac上流畅运行，实现**100%本地语音转文字**；配合**Vision框架截图OCR**和**剪贴板历史追踪**，可构建完整的上下文感知智能输入法系统。豆包输入法的"基础/智能"双模式本质上是**小模型本地ASR + 大模型云端LLM校正**的架构，可通过**MLX本地LLM + 云端API兜底**的方式复现。按住说话（Push-to-Talk）功能已有**VocaMac、VoiceInput、Koe**等多个成熟开源实现可供参考。

## 1. 开源Mac输入法引擎调研：Rime（鼠须管）为什么是最佳选择

### 1.1 Rime/librime 架构深度解析

Rime（中州韵输入法引擎）是macOS平台上最为成熟、扩展性最强的开源输入法框架，其macOS前端实现为**Squirrel（鼠须管）**。从架构层面分析，Rime采用了高度模块化的C++引擎设计，核心库`librime`通过清晰的API接口与前端应用交互，这种分层架构使得语音输入功能的集成完全可行。[^37^] 引擎内部由四大核心模块组成：**配置模块**负责输入方案的解析和管理，支持基于YAML的领域特定语言（DSL）来定义输入法行为；**字典模块**提供词汇检索和用户词典功能，支持百万级词库；**齿轮模块**包含各种翻译器和过滤器，是功能扩展的核心；**杠杆模块**则提供用户配置和部署工具。[^38^]

从技术扩展性的角度来看，Rime最具价值的特性是其**插件系统**。`librime`通过`plugins`模块支持动态扩展，开发者可以创建独立插件作为共享库动态加载，而无需修改核心代码。`librime-lua`插件尤其关键，它将RIME的C++ API以Lua对象的形式暴露出来，允许用Lua脚本扩展处理程序、分词器、翻译器和过滤器。[^45^] 这意味着我们完全可以编写一个Lua插件，在Rime内部调用外部语音转文字服务，将识别结果注入到输入流程中。LuaJIT的运行时性能足以支撑这种集成，且脚本可以动态加载，无需重新编译整个输入法。[^50^]

Rime对**双拼方案的支持**是其另一个核心优势。通过拼写代数（Spelling Algebra）机制，Rime可以创建任意变体拼写规则，内置支持小鹤双拼、自然码、智能ABC、紫光、微软、搜狗等7种主流双拼方案。[^9^] 万象拼音（Wanxiang Pinyin）等第三方方案更是在此基础上增加了直接辅助码、间接辅助码、反查、造词等高级功能，支持9键、14键、18键、26键等多种键盘布局。[^6^] 用户通过`Control+``或F4键即可在不同方案间快速切换，这种灵活性是其他任何开源输入法无法比拟的。

### 1.2 语音输入与Rime的集成路径

Rime本身是一个**纯文本输入法引擎**，不直接处理音频数据。但这并不意味着无法实现语音输入——恰恰相反，Rime的架构设计为语音集成提供了多条清晰的路径。**第一条路径**是通过`librime-lua`插件，在Lua脚本中调用外部语音识别服务（如本地运行的Qwen3-ASR或Apple Speech），将识别结果作为"模拟按键"注入到输入流程中。[^53^] 这种方式的优势在于完全在Rime生态内运行，用户无需切换输入法即可获得语音输入能力。**第二条路径**是开发一个独立的macOS辅助应用（类似VocaMac或VoiceInput），通过辅助功能权限直接向光标处注入文字，与Rime并行工作。[^71^] 这种方式实现更简单，且可以拥有独立的UI和设置界面。

从工程实践的角度，**第二条路径更为推荐**。原因有三：首先，语音识别涉及音频采集、模型推理、LLM校正等多个重计算环节，独立应用可以更好地管理这些资源和生命周期；其次，截图OCR、剪贴板历史等附加功能天然需要一个独立的宿主应用；最后，Swift + SwiftUI的开发体验远优于Lua脚本，且可以充分利用Apple Silicon的硬件加速能力。因此，**推荐的技术架构是：Rime负责拼音/双拼等传统输入，独立语音助手应用负责语音输入和AI增强**，两者通过系统剪贴板和辅助功能API协同工作。

## 2. 语音转文字（ASR）引擎选型：本地优先的模型对比

### 2.1 Qwen3-ASR：最适合中文场景的本地ASR

**Qwen3-ASR**是阿里云通义千问团队推出的开源语音识别模型，是当前**中文语音识别场景下本地部署的最佳选择**。该模型提供了两个主要版本：0.6B（6亿参数）和1.7B（17亿参数），均支持52种语言和22种中文方言，包括粤语、四川话、上海话等。[^84^] 两个版本都支持MLX框架，可在Apple Silicon上通过Metal加速实现高效推理。

从硬件适配的角度分析，**Qwen3-ASR-0.6B是32GB Mac的最佳选择**。该模型FP16精度下模型大小约1.2GB，显存占用约2.1GB；通过MLX框架进行4-bit量化后，**显存占用可降至约500MB**。[^43^] 这意味着即使在运行多个应用的同时，也可以轻松加载该模型。推理速度方面，0.6B的实时因子（RTF）约为0.0094，即处理100秒音频仅需约0.94秒，远超实时需求。[^81^] 在M2 Max芯片上，配合MLX的GPU加速，ASR处理的实时因子可达0.06，100秒音频6秒即可完成识别。[^66^]

| 对比维度 | Qwen3-ASR-0.6B | Qwen3-ASR-1.7B |
|---------|---------------|---------------|
| **参数量** | 6亿 | 17亿 |
| **FP16模型大小** | ~1.2 GB | ~3.4 GB |
| **FP16显存占用** | ~2.1 GB | ~4.8 GB |
| **4-bit量化显存** | **~0.5 GB** | ~1.8 GB |
| **推理速度(RTF)** | 0.0094 | 0.015 |
| **LibriSpeech WER** | 6.8% | 5.2% |
| **最低显存要求** | ≥3 GB | ≥6 GB |
| **支持语言/方言** | 52种 / 22种 | 52种 / 22种 |
| **MLX框架支持** | ✅ | ✅ |
| **适合场景** | 实时转录、资源敏感 | 高精度专业场景 |

对于用户的32GB Mac配置，**同时运行Qwen3-ASR-0.6B和一个4-bit量化的7B级别LLM（如Qwen2.5-7B或GLM-4-9B）完全在可行范围内**。0.6B ASR占用约2GB统一内存，7B LLM 4-bit量化后约占用4-5GB，两者合计约6-7GB，剩余25GB+可供系统和应用使用，多任务运行无压力。

### 2.2 WhisperKit：Apple生态原生ASR方案

**WhisperKit**是Argmax公司开发的Swift语音识别框架，专为Apple Silicon优化，基于OpenAI的Whisper模型。该框架的核心优势在于**原生Apple加速**：通过CoreML将模型编译为Apple Neural Engine（ANE）格式，实现低功耗、高效率的本地推理。[^20^] WhisperKit支持从Tiny到Large v3的5种模型规格，开发者可以根据设备性能和精度需求灵活选择。

| 模型 | 参数量 | CoreML大小 | 速度 | 质量 | 推荐场景 |
|------|--------|-----------|------|------|----------|
| **Tiny** | 39M | ~76MB | ⚡⚡⚡⚡⚡ | 良好 | 快速草稿、低延迟 |
| **Base** | 74M | ~147MB | ⚡⚡⚡⚡ | 更好 | 8GB Mac日常 |
| **Small** | 244M | ~487MB | ⚡⚡⚡ | 很好 | 16GB Mac平衡 |
| **Medium** | 769M | ~1.5GB | ⚡⚡ | 极佳 | 24GB Mac高精度 |
| **Large v3** | 1550M | ~3.1GB | ⚡ | 最佳 | 32GB Mac专业级 |

在M2 Ultra芯片上，WhisperKit运行Large v3 Turbo模型时，GPU+ANE配置可达**72倍实时速度**，ANE单独配置也能达到42倍实时速度。[^28^] 这种性能对于语音输入法场景来说绰绰有余。WhisperKit的**流式识别**能力（通过`TranscriptionSegment`实时回调）尤其适合按住说话的场景，可以在用户说话的同时实时显示识别结果，松手时即可完成最终校正。[^29^]

需要注意的是，**WhisperKit的CoreML编码器会长时间占用Neural Engine**，可能导致与其他依赖ANE的功能（如Face ID、图像识别）产生资源冲突。[^27^] 解决方案是采用**MLX + CoreML双框架调度策略**：小型模型（如VAD、关键词检测）运行在CoreML/ANE上，大型模型（如ASR、LLM）运行在MLX/GPU上，两者互不干扰。[^66^]

### 2.3 Apple Speech框架：零配置的端侧识别

从macOS 14（Sonoma）开始，Apple在设备端内置了**Speech框架**的离线识别能力，无需下载任何模型即可使用。这是实现语音输入功能的**最快路径**——VoiceInput项目就是基于该框架，实现了按住Fn说话、松手注入文字的完整流程，安装包仅约15MB。[^71^] 然而，Apple Speech的识别精度**远低于Qwen3-ASR和Whisper**，特别是在专业术语、中英文混合、方言等场景下。因此，推荐将其作为**fallback方案**：当用户未配置其他ASR引擎时自动启用，确保基础功能可用。

## 3. 按住说话（Push-to-Talk）功能实现分析

### 3.1 交互范式：为什么Push-to-Talk是最优解

调研发现，所有优秀的语音输入法产品都采用**Push-to-Talk（按住说话）**作为核心交互方式，而非传统的"点击开始-再点击结束"模式。这种设计的本质原因是**降低了用户的认知负担和操作失误率**。当用户按住一个键说话时，只需关注"说什么"，无需担心"何时停止"——松开即结束，符合人类自然的语言表达习惯。VocaMac使用**右Option键**作为默认触发键，VoiceInput使用**Fn键**或**右Command键**，两者都支持双击切换为锁定模式（按一次开始，再按一次结束）。[^1^] [^71^]

从技术实现角度，Push-to-Talk需要**全局热键监听**能力。在macOS上，这通过`CGEventTap`（Core Graphics事件钩子）实现，需要**辅助功能（Accessibility）权限**和**输入监控（Input Monitoring）权限**。[^71^] 当检测到按键按下时启动音频录制，检测到按键释放时停止录制并触发识别流程。整个过程的延迟（从松手到文字出现）可以控制在**1-3秒**内，取决于ASR模型的速度和是否启用LLM校正。

### 3.2 开源实现参考与架构设计

**VocaMac**是基于WhisperKit的Push-to-Talk实现的最佳参考。该项目使用纯Swift开发，架构清晰：**全局热键模块**通过`CGEventTap`监听右Option键；**音频录制模块**使用`AVAudioEngine`捕获PCM音频流；**语音识别模块**通过WhisperKit进行CoreML推理；**文字注入模块**使用Apple Events将识别结果插入到当前光标位置。[^1^] 其双模式设计（按住模式 + 双击切换模式）值得直接借鉴。

**VoiceInput**则展示了更轻量级的实现路径。该项目仅用**一个Prompt通过Claude Code生成**，功能却相当完整：按住Fn/右⌘录音，悬浮波形HUD显示实时状态，松手后先通过Apple Speech本地识别，再可选地通过OpenAI兼容API进行LLM纠错，最后通过"保存剪贴板→写入文本→Cmd+V粘贴→恢复剪贴板"的四步注入流程将文字插入目标应用。[^71^] 其中**中文输入法兼容处理**（注入前自动退出拼音输入法，注入后恢复）是容易被忽略但至关重要的细节。

推荐采用的**状态机设计**如下：

```
[Idle空闲] --(按住热键)--> [Recording录音中] --(实时显示波形和 interim 结果)-->
[Recording录音中] --(松开热键)--> [Recognizing识别中] --(ASR推理)-->
[Recognizing识别中] --(LLM校正开启)--> [Refining优化中] --(LLM推理)-->
[Refining优化中] --(完成)--> [Injecting注入中] --(Cmd+V)--> [Idle空闲]
```

## 4. 大小模型切换机制：复现豆包输入法的"基础/智能"双模式

### 4.1 豆包输入法的技术架构拆解

豆包输入法采用**Seed-ASR 2.0**作为语音识别引擎，这是字节跳动自研的端到端语音模型。其最核心的架构创新是**"基础输入"模式与完整模式的双层设计**：基础模式下仅使用本地**150MB的离线语音模型**进行纯ASR转写，不联网，不使用任何AI增强功能；完整模式下则启用云端大模型进行上下文感知识别和LLM校正。[^89^] 这种模式切换的本质是**在隐私/速度/精度之间做权衡**——基础模式适合敏感场景（如输入密码），完整模式适合日常交流。

从技术实现角度分析，豆包的"基础/智能"双模式可以被理解为以下架构：

| 模式 | ASR引擎 | LLM校正 | 网络依赖 | 隐私级别 | 适用场景 |
|------|---------|---------|----------|----------|----------|
| **基础模式** | 本地小模型(150MB) | ❌ | 不需要 | 极高 | 离线环境、敏感输入 |
| **智能模式** | 云端Seed-ASR 2.0 | ✅ 云端大模型 | 需要 | 中 | 日常输入、专业术语 |

豆包输入法的另一个关键技术点是**ASR的上下文感知能力**。传统ASR是"听见什么写什么"，而Seed-ASR 2.0通过深度理解上下文完成更精准的识别，专有名词、多音字、人名地名的召回率有显著提升。[^34^] 这正是用户提到的"Claude code不会被识别为clock"的核心能力——ASR需要具备**语言模型的推理能力**，而不是单纯的声学匹配。

### 4.2 本地复现方案：MLX小模型 + 云端API兜底

要复现豆包的双模式架构，推荐采用以下**分层策略**：

**第一层：本地MLX ASR（基础模式）**
- 使用**Qwen3-ASR-0.6B（4-bit量化）**作为默认本地ASR引擎
- 显存占用仅~500MB，推理速度极快（RTF 0.0094）
- 支持52种语言和22种方言，涵盖绝大多数使用场景
- 100%离线运行，零隐私风险

**第二层：本地MLX LLM校正（增强模式）**
- 使用**Qwen2.5-3B-Instruct（4-bit量化）**或**Qwen3-4B-4bit**进行文本校正
- 显存占用约2-3GB，可处理同音字纠错、专业术语修正、口水词过滤
- 4-bit量化后的7B模型（如GLM-4-9B-4bit）可实现更强的理解和改写能力，显存约5-6GB
- 完全本地运行，适合处理敏感内容

**第三层：云端API兜底（智能模式）**
- 当本地模型识别置信度低于阈值时，自动调用云端ASR API（如通义千问、豆包、OpenAI Whisper API）
- 当需要更强的文本理解和生成能力时，调用云端大模型API
- 用户可配置API Key和优先级策略

**大小模型切换的触发策略**：
- **手动切换**：用户通过快捷键或菜单栏在"基础/智能"模式间切换
- **自动切换**：根据网络状态自动选择本地/云端模型
- **置信度触发**：当本地ASR的输出置信度低于设定阈值（如0.85）时，自动调用云端ASR进行复核
- **内容感知**：当检测到专业术语、外语混合、代码等复杂内容时，自动启用大模型校正

## 5. LLM文本校正与上下文感知：从"转写"到"理解"

### 5.1 为什么需要LLM校正

语音识别（ASR）的本质是**将声学信号映射为文字序列**，但它不理解文字的含义。这就是为什么用户提到"Claude code会被识别为clock"——ASR从声学角度判断"clock"是最可能的词，但缺乏上下文推理能力来判断这里应该是"Claude"。**LLM校正（LLM Refinement）**的核心价值就在于弥补这一鸿沟：让语言模型理解用户的真实意图，对ASR的原始输出进行语义层面的修正。[^71^]

豆包输入法的LLM校正能力包括：**同音字纠错**（如"代码"vs"带妈"）、**专业术语修正**（如"Claude code"vs"clock"）、**智能标点添加**、**口水词过滤**（删除"嗯"、"啊"、"那个"等 filler words）、**语句通顺化**（调整语序使表达更自然）。这些能力需要一个**7B级别以上的大模型**才能较好地完成，3B模型可以处理基础纠错，7B+模型才能实现深度理解。[^70^]

### 5.2 上下文感知的三重维度

要实现真正智能的语音输入，系统需要维护**三重上下文**：

**第一重：屏幕视觉上下文（截图OCR）**
- 当用户激活语音输入时，系统自动截取当前屏幕
- 使用**Vision框架的VNRecognizeTextRequest**进行本地OCR
- 提取屏幕上的文字内容作为LLM的上下文提示
- 例如：用户在VS Code中选中代码后语音说"解释一下这段"，系统通过截图OCR获取代码内容，LLM据此生成解释
- Vision框架的OCR完全本地运行，无需网络，支持中文和英文

**第二重：剪贴板历史上下文**
- 维护最近N条（如50条）剪贴板记录
- 包括文本、代码片段、链接等内容
- 作为LLM的short-term memory，帮助理解用户的当前工作主题
- 例如：用户之前复制了"Kubernetes deployment yaml"，随后语音说"帮我写个service"，系统知道上下文是K8s

**第三重：会话历史上下文**
- 维护当前应用/当前文档的输入历史
- 作为LLM的long-term context，保持领域一致性
- 例如：用户在写技术博客，之前的输入都是关于"Rust async/await"的内容，后续的语音输入会保持技术语境

### 5.3 推荐的开源LLM模型

| 模型 | 量化版本 | 显存占用 | 中文能力 | 速度 | 推荐场景 |
|------|----------|----------|----------|------|----------|
| Qwen2.5-3B-Instruct | 4-bit | ~2GB | 强 | 快 | 基础文本清洗 |
| **Qwen3-4B-4bit** | **4-bit** | **~2.5GB** | **很强** | **快** | **平衡选项（推荐）** |
| Qwen3-8B-4bit | 4-bit | ~5GB | 很强 | 中 | 深度改写 |
| **GLM-4-9B-4bit** | **4-bit** | **~5.5GB** | **极强** | **中** | **中文场景首选** |
| Llama-3.2-3B-Instruct-4bit | 4-bit | ~2GB | 中等 | 快 | 英文优先 |

对于32GB Mac用户，**推荐同时部署Qwen3-ASR-0.6B（~2GB）和GLM-4-9B-4bit（~5.5GB）**，合计约7.5GB统一内存，剩余24GB+供系统和应用使用。如果追求更低资源占用，**Qwen3-4B-4bit是最佳平衡点**，文本校正能力足以应对大多数场景。

## 6. 截图OCR与视觉上下文实现方案

### 6.1 macOS屏幕捕获技术选型

在macOS上获取屏幕内容有两种主要方式：**CGWindowListCreateImage**（传统API）和**ScreenCaptureKit**（现代API）。ScreenCaptureKit是Apple在macOS 12.3+推出的框架，提供了更高效、更灵活的屏幕捕获能力，支持窗口级、应用级和显示器级的捕获，且性能开销更低。[^62^] 对于语音输入法场景，推荐在激活语音输入时，使用**ScreenCaptureKit捕获当前最前方应用窗口的内容**，而非整个屏幕，以减少不必要的隐私风险和OCR处理量。

对于OCR识别，**Vision框架**是最佳选择。`VNRecognizeTextRequest`支持中文和英文的本地文字识别，完全离线运行，无需网络请求。在macOS 13+上，Vision框架的OCR精度已接近商业级水平，对于代码、文档等常见场景的识别准确率非常高。[^61^] 识别后的文字内容可以作为`system prompt`的一部分传递给LLM，让模型"看到"用户当前在做什么，从而生成更准确的输入内容。

### 6.2 视觉上下文的Prompt工程

将截图OCR结果融入LLM的Prompt需要精心设计。推荐的Prompt模板如下：

```
你是一个智能语音输入法助手。用户正在使用语音输入，你需要根据以下上下文理解用户的意图，
对语音识别结果进行校正和优化。

【屏幕当前内容】
{{ ocr_text }}

【用户最近复制的内容】
{{ clipboard_history }}

【语音识别原始结果】
{{ asr_raw_text }}

请根据屏幕内容和剪贴板历史，判断用户的领域和意图，然后：
1. 修正语音识别错误（特别是同音字、专业术语）
2. 删除口水词（嗯、啊、那个等）
3. 优化语句通顺度
4. 保持代码、技术术语的准确性
5. 如果用户说"解释这段代码"、"优化这个函数"等指令，请结合屏幕内容给出详细回答

只输出最终文字，不要添加解释。
```

这种上下文注入方式使得语音输入法从"单纯的转写工具"升级为"理解上下文的智能助手"，大幅提升了在专业场景下的实用性。

## 8. 关键结论与建议

### 8.1 调研核心发现

通过对**VocaMac、VoiceInput、Koe、type4me、PushToTalk、voxt**等6个开源语音输入项目的深度调研，以及对**Qwen3-ASR、WhisperKit、Apple Speech**等3个ASR引擎的全面对比，得出以下核心结论：

**第一，32GB Mac完全可以本地运行完整的语音输入法系统**。Qwen3-ASR-0.6B 4-bit量化后仅占用约500MB显存，Qwen3-4B-4bit约2.5GB，两者合计3GB左右，远低于32GB的容量上限。用户甚至可以同时加载更大的LLM模型（如GLM-4-9B-4bit，~5.5GB）来获得更强的文本理解能力。

**第二，豆包输入法的"基础/智能"双模式本质是"小模型本地ASR + 大模型云端/本地LLM"的分层架构**。这一架构完全可以本地复现：Qwen3-ASR作为本地小模型处理基础识别，MLX本地LLM处理文本校正，云端API作为可选增强。用户可根据隐私需求和场景灵活切换。

**第三，截图OCR和剪贴板历史是实现"上下文感知"的关键技术**。Vision框架提供高精度本地OCR，ScreenCaptureKit提供高效截图，NSPasteboard提供剪贴板访问——三者结合可以让语音输入法"看到"用户的屏幕内容，从而理解用户的真实意图。

### 8.2 推荐的最小可行产品（MVP）

对于快速验证方案可行性，推荐以下**最小组合**：

- **Rime（鼠须管）**：负责所有拼音/双拼输入，已在macOS上成熟运行
- **VoiceInput**：作为语音输入端，按住Fn说话、松手注入，基于Apple Speech零配置运行
- **后续升级**：替换ASR为Qwen3-ASR（MLX），添加本地LLM校正，集成截图OCR

这个MVP可以在**1-2天**内部署完成，立即获得Push-to-Talk语音输入能力，后续逐步替换和增强各个模块。
